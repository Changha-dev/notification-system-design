## 프로젝트 개요
다양한 이벤트 알림 발송 시의 상황을 가정하고 백엔드단 시스템 설계 및 구현을 하였습니다.  
멱등성, 예외처리에 대한 재시도, 그리고 최대한 기존 기술 스택을 활용한 아키텍처 설계 고민을 하였습니다.(추가적인 인프라 발생 비용을 줄이기 위해)
![프로젝트 전체 아키텍처](./system-architecture.png)
전체 아키텍처는 위와 같습니다.
크게 3가지 기능을 구현하였습니다.
- 중복처리에 대한 **멱등성 처리**
- 알림 이벤트에 대한 영속성을 관리하기 위한 **트랜잭션 아웃박스 패턴**
- 처리량을 향상시키기 위한 **비동기 스레드 도입**

## 기술 스택

## 요구사항 해석 및 가정
```plain
배경 시나리오 1)
- 알림 처리 실패가 비즈니스 트랜잭션에 영향을 주어서는 안 됩니다. 단, 예외를 단순히 무시하는 방식으로 이를 달성해서는 안 됩니다.
- 네트워크 장애, 외부 이메일 서버 오류 등 일시적 장애에 대비해 재시도가 가능해야 합니다. 
```
트랜잭션 분리로 알림 처리를 독립적으로 가능하도록 해야 되겠다고 생각이 들었습니다.  
예외를 무시하지 않는 것과 일시적 장애 케이스들은 알림 이벤트 영속성을 보장하며 재시도 전략을 통해 완료되도록 설계를 생각하였습니다.

```plain
배경 시나리오 2)
- 동일한 이벤트에 대해 알림이 중복 발송되면 안 됩니다.
```
**멱등 키**를 설계하여 중복 요청을 방지하도록 해야되겠다고 생각이 들었습니다. 

```plain
배경 시나리오 3)
- 이벤트 발생 시 사용자에게 이메일 또는 인앱 알림을 발송해야 합니다.
```
이 부분은 이메일 또는 인앱 등 추후에 확장 가능성을 고려한 **전략 패턴**이 적합하겠다고 판단이 들었습니다.  

```plain
필수 구현 1. 알림 발송 요청 API
- 사용자 알림 목록 조회: 수신자 기준 알림 목록(읽음/안읽음 필터 포함)
```
읽음/안읽음 부분은 **복합 인덱스**를 통해 조회 성능 향상을 생각하였습니다.

```plain
필수 구현 2. 알림 처리 상태 관리
- 알림 상태를 적절히 정의하고, 각 상태의 전이 조건을 설계하세요.
- 발송 실패 시 재시도 정책과 최종 실패 처리 방식을 설계하세요.
- 실패 사유는 기록되어야 합니다.
```
알림 발송 상태는 PENDING, SENDING, SENT, FAILED로 구상하였습니다.  
각 상태 전이 과정이 언제 발생하는지와 예외 케이스일 경우 어떻게 대처할 건지 고민이 필요했습니다. 

```plain
필수 구현 5. 운영 시나리오 대응
- 처리 중 상태가 일정 시간 이상 지속되는 경우 복구 방법을 설계하세요.
- 서버 재시작 후에도 미처리 알림이 유실 없이 재처리되어야 합니다.
- 다중 인스턴스 환경에서도 동일 알림이 중복 처리되어서는 안 됩니다.
```
- 처리 중 상태일 때 어느 부분에서 문제가 발생했는 지 파악하는 것이 필요할 것이라고 생각이 들었습니다.
- 서버 재시작 후에도 미처리 알림이 없기 위해 알림 이벤트를 Outbox 패턴을 활용한 테이블에 기록하는 것이 필요하다고 판단하였습니다.
- 다중 인스턴스 환경에서 동일 알림 중복 처리를 막기 위해 멱등성 개념과 더불어 Outbox 테이블에 동시에 접근하는 것을 막는 기법이 필요함을 느꼈습니다.

```plain
선택 구현
- 발송 스케줄링: 특정 시각에 발송 예약 기능
- 알림 템플릿 관리 (타입별 메시지 템플릿)
- 읽음 처리: 여러 기기에서 동시에 읽음 처리 요청이 오면 어떻게 처리할 것인가?
- 최종 실패 알림 보관 및 수동 재시도: 재시도 시 재시도 횟수 초기화 여부를 어떻게 정책화할 것인가?
```
- 발송 스케줄링은 스케줄러를 이용하여 손쉽게 구현 할 수 있지만 싱글 스레드로 동작한다는 점과 멀티 인스턴스에서 중복되는 경우 고려하는 것이 필요하다고 생각하였습니다.
- 알림 템플릿은 다양한 이벤트 타입별 분리와 재사용성 가능한 코드를 설계를 신경써야겠다고 생각하였습니다.
- 여러 기기에서 동시에 읽음 처리 요청이 왔을 때는 읽음 처리의 카운트를 관리하는지 유무에 따라 다르게 처리가 필요할 것 같습니다.
  - **관리하지 않는 경우**: 중복 처리가 상관 없음
  - **관리하는 경우**: 중복처리하면 카운트 정합성이 깨짐
- 재시도: 고민..

## 설계 결정과 이유
### 1. 중복 요청 방지
**상황**: `클라이언트 -> 스프링 서버`, `스프링 서버 -> DB`, `스프링 서버 -> 외부 서버` 구간에서 재시도나 중복 호출이 발생할 수 있습니다.  
**문제점**: 동일한 이벤트에 대해 알림이 여러 번 생성되거나 실제 발송이 중복될 수 있어 요구사항을 만족하지 못합니다.  
**해결책**:
- 이벤트 기준 멱등 키를 생성하고 DB 유니크 제약을 통해 동일 알림이 한 번만 생성되도록 설계하였습니다. 
- 또한 Outbox 처리 시에도 하나의 레코드를 한 번에 하나의 워커만 선점하도록 하여 다중 인스턴스 환경에서도 중복 발송을 방지합니다.

### 2. 비즈니스 트랜잭션과 알림 전송 분리
**상황**: 알림 전송 로직이 비즈니스 로직과 같은 트랜잭션 안에서 처리될 수 있습니다.  
**문제점**: 비즈니스 트랜잭션이 롤백되어도 사용자는 이미 알림을 받는 문제가 생길 수 있습니다. 반대로 알림 실패가 비즈니스 로직까지 실패시키는 구조도 요구사항에 맞지 않습니다.  
**해결책**: 
- Notification과 Outbox를 같은 트랜잭션에서 함께 저장하고, 실제 발송은 비동기 워커가 별도 트랜잭션으로 처리하도록 분리하였습니다. 
- `@TransactionalEventListener(phase = AFTER_COMMIT)`는 필요 시 보조 트리거로만 활용하고, 핵심 내구성 보장은 Outbox 패턴으로 처리합니다.

### 3. 네트워크 장애와 재시도 처리
![네트워크 에러 상황](./network-exception.png)

#### 3.1 Spring <-> 외부 서버 구간 장애
**상황**: `Spring <-> 외부 서버` 구간에서 connection timeout, read timeout, 429, 일시적 5xx 오류가 발생할 수 있습니다.  
**문제점**: 일시적인 오류만으로 즉시 최종 실패 처리하면 복구 가능한 알림까지 유실될 수 있고, 반대로 무제한 재시도는 자원 낭비와 장애 전파를 유발할 수 있습니다.  
**해결책**:
- 재시도 가능한 예외에 한해 `retryCount`, `nextRetryAt` 기반의 지수 백오프 정책을 적용하였습니다.
- 최대 재시도 횟수를 초과하면 최종 실패 상태로 전환하고, 실패 사유를 기록하여 운영자가 원인을 추적할 수 있도록 설계하였습니다.
- read timeout처럼 전송 성공 여부가 불명확한 경우를 고려하여 멱등 키 기반 중복 방지 전략을 함께 적용하였습니다.

#### 3.2 Spring <-> DB 구간 장애
**상황**: `Spring <-> DB` 구간에서 connection 문제나 상태 갱신 실패가 발생할 수 있습니다.  
**문제점**: 
- 송신 실패: 알림 요청 접수 자체가 실패할 수 있습니다.
- 수신 실패: 외부 발송 후 DB 상태 반영이 실패하면 실제 발송 결과와 내부 상태가 불일치할 수 있습니다.
**해결책**:
- 송신 해결: 
  - 알림 요청 접수는 Notification과 Outbox를 같은 트랜잭션으로 저장하여, 영속화 성공 시에만 요청이 정상 접수된 것으로 간주하였습니다.
  - 비동기 처리 중 DB 갱신 실패가 발생하더라도 Outbox를 기준으로 다시 조회 및 재처리할 수 있도록 설계하였습니다.
- 수신 해결:
  - 동일 요청 재시도를 하더라도 만약 성공이 되었다면 중복되지 않게 멱등 키를 정의하고, 유니크 제약을 적용하여 중복 생성을 방지하였습니다.

### 4. 서버 재시작 및 처리 중단 복구
**상황**: 알림 처리 도중 서버가 종료되거나 워커 스레드가 비정상 종료될 수 있습니다.  
**문제점**: 메모리 기반 큐만 사용할 경우 미처리 알림이 유실될 수 있고, `SENDING` 상태로 남은 작업이 영구적으로 정체될 수 있습니다.  
**해결책**: 
  - Outbox를 DB에 영속 저장하여 서버 재시작 후에도 미처리 작업을 다시 조회할 수 있도록 하였습니다. 
  - 또한 `lockedAt` 또는 lease 기반 복구 정책을 두어 일정 시간 이상 처리 중 상태로 남은 작업은 재처리 가능하도록 설계하였습니다.

### 5. 채널 확장성과 실행 모델 교체 가능성
**상황**: 현재는 EMAIL, IN_APP 두 채널만 요구되지만 이후 SMS, PUSH 등으로 확장될 수 있고, 발송량 증가에 따라 실행기 구성도 바뀔 수 있습니다.  
**문제점**: 채널별 로직과 실행 정책이 서비스 내부에 강하게 결합되면 기능 추가나 운영 환경 변경 시 수정 범위가 커집니다.  
**해결책**: 
- 채널별 발송 로직은 전략 패턴으로 분리하고, 실행기는 스프링 빈으로 주입받아 platform thread 기반 executor와 virtual thread 기반 executor를 교체 가능하도록 설계하였습니다. 
- 실제 메시지 브로커 도입 시에도 발송 파이프라인을 크게 바꾸지 않도록 구조를 분리하였습니다.
